이 문서에서는 애플리케이션 성능 장애 유형 중 하나인 **HTTPC 지연**(HTTP 호출 지연)을 모니터링 관점에서 소개하려 합니다.

먼저 AI 히트맵 패턴 알림을 통해 이상 현상을 인지했다고 가정 후 ***애플리케이션 대시보드***의 ***히트맵*** 위젯에서 에러 추정 구간을 바로 드래그해 빠르게 ***트랜잭션 분석***창을 확인하겠습니다. 그리고 확인한 시간대를 특정해 ***통계*** 메뉴와 ***트랜잭션 검색*** 메뉴로 이동 후 트레이스를 분석해 장애 원인을 추론하는 과정을 안내하겠습니다.

:::note

**미리 알아보기**

AI 히트맵 패턴 알림 → ***애플리케이션 대시보드*** → ***트레이스 분석*** → ***통계*** → ***트랜잭션 검색*** → ***트레이스 분석***

* 이상 패턴 알림을 통해 문제 현상을 인지 후 ***애플리케이션 대시보드***를 확인합니다.

* ***애플리케이션 대시보드 히트맵*** 위젯에서 문제 추정 구간을 바로 드래그해 ***트랜잭션 분석***창을 조회합니다.

* 트랜잭션 ***경과 시간***을 확인해 에러 상황을 특정합니다. 경과 시간 대비 HTTP 호출에 소요되는 시간이 높은 것을 확인합니다.

* ***통계*** 메뉴의 ***HTTP 호출*** 탭으로 이동해 문제가 되는 특정해 트랜잭션을 확인합니다.

* ***트랜잭션 검색*** 메뉴로 이동해 에러 발생건 중 ***액티브 스택***이 확보된 트랜잭션을 확인합니다.

## AI 기반 히트맵 이상 패턴 알림

장애 현상을 인지하고 일차적인 분석을 통해 장애 원인을 추적하는 과정은 신속함이 핵심입니다. 또한 해당 장애를 인지한 시점의 운영 담당자가 빠르게 조처할 수 있도록 직관적인 모니터링이 가능해야 합니다. 

그렇다면 이러한 이상 현상 인지와 동시에 진행되는 일차 분석 시 필요한 신속성과 직관성을 와탭은 어떻게 제공하고 있을까요? 이 문서에서 AI 기반의 히트맵 패턴 알림을 통해 성능 분석 숙련자가 아닌 신규 운영자가 특정 장애, 즉 HTTPC 지연 장애를 인지하고 추적하는 과정을 예시로 살펴보겠습니다. 

알림 설정의 경우 모니터링 과정에서 매우 중요한 단계입니다. 일반적으로 숙련자가 운영 경험을 바탕으로 사용자 환경에 따라 여러 알람 조건을 지정합니다. 하지만 **히트맵 패턴 알림**의 경우 번거로운 세부 설정 과정이 필요하지 않습니다. 

<ImgLang img='best-p/about-apm-httpc-hitmap-pattern-event-setting.png' desc='AI 히트맵 이상 패턴 알림' /> 

***이벤트 설정*** 메뉴에서 각 이상 패턴 알림을 활성화하면 수많은 히트맵 패턴 학습을 바탕으로 사용자의 실시간 수집 데이터에서 유사한 패턴 발생 시 즉각적인 알림을 제공합니다. 클릭 한 번만으로 사용자는 이상 현상 발생 시 빠르게 해당 이슈를 인지하고 이에 대응할 수 있게 됩니다.

사용자는 하루종일 모니터링 화면을 응시하는 대신 이메일, SMS 등 다양한 경로로 설정해 둔 와탭 알림을 통해 다음과 같이 현재 발생한 문제 상황을 빠르게 파악할 수 있습니다. 예시의 세로 패턴(Vertical pattern) 메일 알림 화면에서 ***와탭에 로그인하기***를 선택해 서비스 화면으로 이동하겠습니다. 

<ImgLang img='best-p/about-apm-httpc-dashboard-w-notice.png' desc='AI 히트맵 이상 패턴 알림과 애플리케이션 대시보드 sc' />

:::note 

와탭은 SaaS 방식으로 모니터링을 제공하기 때문에 다양한 서비스와 환경에 대해 패턴별로 수많은 케이스를 수집할 수 있습니다. 특허를 취득한 와탭 **AI 기반의 히트맵 패턴 경고 알림**에 대한 자세한 내용은 [다음 문서](https://docs.whatap.io/java/hitmap-notice)를 참조하세요. 

:::

## 애플리케이션 대시보드

알림 시점에 서비스 화면으로 즉시 이동 시 ***애플리케이션 대시보드***를 조회할 수 있습니다. ***애플리케이션 대시보드***에서 조회 가능한 최근 10분 이상의 시간이 경과한 경우 ***분석*** 메뉴 하위의 ***히트맵***으로 이동합니다. 여기서는 전자의 동선으로 진행해 보겠습니다.

<ImgLang img='best-p/about-apm-httpc-dashboard.png' desc='애플리케이션 대시보드' /> 

먼저 ***애플리케이션 대시보드*** 상단에서 에이전트 상태를 확인하겠습니다. 해당 프로젝트는 총 3개의 에이전트가 활성화된 상태입니다. ***히트맵*** 위젯을 살펴보면 **세로 패턴**이 반복적으로 나타나고 있습니다. 세로 패턴은 일시적으로 락이 발생해 요청건에 대한 처리를 대기하다 락의 해소로 대기 중이던 트랜잭션이 비슷한 시간대에 한 번에 종료해 만들어지는 패턴입니다. 패턴 발생 시점에 ***액티브 스테이터스*** 내 ***HTTPC*** 수치의 증가, ***Apdex*** 하락, ***평균 응답시간***의 급격한 상승 등을 확인할 수 있습니다.

* ***액티브 스테이터스***: 액티브 트랜잭션 수와 진행 상태를 5초마다 수집한 차트입니다.

* ***Apdex***: 사용자 만족도 판단에 활용할 수 있는 애플리케이션 성능 지표입니다.

    * 0.0~0.7(<span class='vslow'>Poor</span>) 

    * 0.7~0.85(<span class='slow'>Fair</span>) 

    * 0.85~0.95(<span class='snormal'>Good</span>)

    * 0.95~1.0(<span class='normal'>Excellent</span>)

* ***평균 응답시간***: 사용자 요청 반환에 걸리는 평균 시간으로 유의미한 튜닝 지표입니다.

***액티브 스테이터스***를 통해 ***HTTPC***의 높은 수치를 확인할 수 있는데 이것으로 외부와 연결된 서버의 응답이 제대로 이루어지지 않고 있음을 추측할 수 있습니다. ***Apdex*** 위젯에서 확인한 62.42라는 수치는 지연으로 업무 처리가 어려운 상태(<span class='vslow'>Poor</span>)이며 이와 같은 ***Apdex*** 하락이 패턴 발생 시점에 되풀이되고 있습니다. ***평균 응답시간***의 경우 패턴 발생 시점에 급격한 상승을 반복하고 있습니다.

다음으로 ***히트맵*** 위젯에서 ![위 화살표 아이콘](/img/ic-s-up.svg) 화살표 아이콘을 선택해 응답지연 구간을 확인 후 에러 추정 구간을 드래그해 보겠습니다. 이때 조회 구간을 좁게 선택하는 것이 상황을 특정하기 용이합니다. 응답시간 지연이 있는 상단의 세로 패턴 드래그 시 다음과 같이 ***트랜잭션 분석*** 창이 나타납니다.

<ImgLang img='best-p/about-apm-httpc-dashboard-tx-trace.png' desc='히트맵 위젯과 트랜잭션 트레이스' /> 

:::note 

* 히트맵 패턴은 다양한 유형을 가지고 있습니다. 세로 패턴 외 다른 유형에 대한 자세한 내용은 [다음 문서](https://docs.whatap.io/best-practice-guides/about-apm-hitmap-class#about-hitmap-pattern-analysis)를 참조하세요.

* ***애플리케이션 대시보드*** 분석에 대한 자세한 내용은 [다음 문서](https://docs.whatap.io/best-practice-guides/about-apm-dashboard)를 참조하세요. 

:::

## 트랜잭션 트레이스 분석 

<ImgLang img='best-p/about-apm-httpc-tx-trace.png' desc='트랜잭션 분석' /> 

***트랜잭션 트레이스 분석*** 창에서 트랜잭션 트레이스 목록을 통해 총 3개의 에이전트 중 특정 에이전트 Test-agent-2에서 에러가 발생하고 있는 것과 다수의 트랜잭션에서 ***경과 시간*** 대비 ***HTTP 호출 시간***이 차지하는 비율이 큰 것을 확인할 수 있습니다. 

예시로 `/test/action.do` 트랜잭션의 경우 총 ***경과 시간*** 26,800ms중 HTTP 호출에만 26,768ms가 소요되었습니다. 또한 목록 왼쪽에서 에러 메시지 `SLOW_HTTPC`를 확인할 수 있습니다. 즉 일차적으로 HTTP 호출에 문제가 있다는 것을 추측할 수 있습니다. 다음으로 해당 트랜잭션을 선택해 상세 정보를 살펴보겠습니다.

:::note 

***에러 메시지*** `java.io.IOException`의 경우 Tomcat 서버 환경에서 발생하는 예외입니다. ***에러 클래스*** `org.apache.catalina.connector.ClientAbortException`로 분류되는데 사용자가 요청 완료 전에 연결을 먼저 끊었을 때 발생합니다. 일반적으로 심각한 문제를 나타내지는 않지만 사용자 경험에 영향을 미칠 수 있습니다.

:::

<ImgLang img='best-p/about-apm-httpc-tx-trace-anl.png' desc='트랜잭션 분석 테이블' /> 

트랜잭션 목록에서 트랜잭션 선택 시 오른쪽 화면에서 해당 트랜잭션의 상세 수행 이력을 스텝별로 확인할 수 있습니다.
